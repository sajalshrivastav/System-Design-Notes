# Critical Rendering Path (CRP)

> **Definition:** The sequence of steps a browser takes to convert HTML, CSS, and JavaScript into pixels on the screen.

---

## What is the Critical Rendering Path?

The **Critical Rendering Path** is the path that a browser takes to render a page on the screen — step by step. It represents the minimum work the browser must do before the user can see anything meaningful on screen.

Optimizing the CRP = reducing the time to first render.

---

## How the Browser Renders a Frame

Rendering is handled by the **Rendering Pipeline** — a series of sequential stages the browser goes through to paint each frame.

### The Rendering Pipeline — Step by Step

```
HTML/CSS/JS
    ↓
1. DOM Construction
    ↓
2. CSSOM Construction
    ↓
3. Render Tree Construction
    ↓
4. Layout (Reflow)
    ↓
5. Paint
    ↓
6. Compositing
    ↓
Pixels on Screen 🖥️
```

---

### Step 1 — DOM Construction
- Browser parses the **HTML** document top to bottom
- Converts HTML tags into **DOM nodes**
- Builds the **Document Object Model (DOM)** tree
- **Blocked by:** `<script>` tags (unless `async`/`defer`)

### Step 2 — CSSOM Construction
- Browser parses all **CSS** (stylesheets, inline, embedded)
- Builds the **CSS Object Model (CSSOM)** tree
- **Render-blocking:** browser won't render until CSSOM is complete
- CSS is treated as a **render-blocking resource**

### Step 3 — Render Tree Construction
- Browser combines **DOM + CSSOM** into a **Render Tree**
- Only includes **visible nodes** (e.g., `display: none` nodes are excluded)
- Each node in the render tree holds its computed visual styles

### Step 4 — Layout (Reflow)
- Browser calculates the **exact position and size** of every element
- Uses the **box model** to determine geometry
- Affected by: viewport size, element dimensions, font sizes
- **Expensive operation** — triggered again if DOM/styles change

### Step 5 — Paint
- Browser fills in **pixels** for each element (colors, text, images, borders, shadows)
- Happens on **layers** (elements can be promoted to their own layer)
- **Repaint** = only visual change, no geometry change (e.g., `color`, `background`)

### Step 6 — Compositing
- Browser takes the painted **layers** and combines them into the final frame
- Happens on the **GPU** (fast ✅)
- Properties like `transform` and `opacity` only trigger compositing — skipping Layout and Paint entirely → **best for animations**

---

## Key Concepts to Remember

| Term | Meaning |
|------|---------|
| **Render-blocking resource** | CSS/JS that pauses rendering until fully loaded |
| **Reflow (Layout)** | Recalculating geometry — expensive |
| **Repaint** | Redrawing pixels — moderate cost |
| **Composite-only** | Only GPU compositing — cheapest (`transform`, `opacity`) |
| **Critical resources** | Resources needed before first render |
| **Critical bytes** | Total bytes needed to complete CRP |

---

## CRP Optimization — Quick Notes

- Minimize render-blocking CSS and JS
- Use `async` / `defer` on `<script>` tags
- Inline critical CSS (above-the-fold styles)
- Reduce critical resource chain depth
- Prefer `transform` / `opacity` for animations (composite-only, no reflow/repaint)
